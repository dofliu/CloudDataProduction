# 05 · 2D 世界與教學系統

## 2D 等距世界(皮)

不做 3D。用 React + PixiJS + Kenney.nl(CC0)等距素材。世界本身**不存任何狀態**,
所有顏色 / 動畫都來自 `/ws/telemetry` 與 `/ws/events`。

### 三層縮放 = 資料層級

```
俯瞰圖(園區地圖)  → 公司 / 廠房      → 設備           → tag
park map            company buildings    device sprites    即時值面板
顯示各公司燈號       點進去看設備佈局     紅綠燈 + 動畫      tag 表 + 即時曲線
```

- **俯瞰**:等距園區,道路、廠房、綠地;每間公司一棟,屋頂燈號彙整該公司最差設備狀態。
- **公司**:點工廠進去,看該公司設備擺放;設備以 sprite 呈現,狀態驅動顏色
  (綠=正常 / 黃=警告 / 橘=**預測故障** / 紅=故障 / 灰=停機)。
- **設備**:點設備開即時面板,列 tag 值 + 近時曲線 + 該設備工單。

### 動畫(氛圍,別過度)

輸送帶滾動、機械手臂擺動、AGV 沿 `pos_x/y` 在地圖上移動、故障設備閃紅、
煙囪冒煙、預測故障橘色脈動。動畫只是把 telemetry 視覺化,做到「有感」即可,別追求寫實。

## 三個面(責任切乾淨)

| 面 | 使用者 | 用途 |
|----|--------|------|
| **教師控制台**(`web/teacher`) | 老師(需 auth) | 上帝視角:疊在世界上,建/改設備、即時注入故障、看 ground-truth health/RUL |
| **參考客戶端儀表板**(`web/teacher`) | 老師 | 標準答案:讀同一場景、自動連全設備輪詢、即時顯示。課堂演示用 |
| **公開設備目錄**(`web/catalog`) | 學生(唯讀) | 規格書 / 接線圖:每台設備連什麼 port/unit_id/topic/node、有哪些 tag |

> 學生**自己開發客戶端**抓取(不是用老師的儀表板)。他們靠設備目錄知道要連什麼、抓什麼。
> 老師左手控制台注入故障、右手參考儀表板看紅燈 —— 課堂高光時刻。

## 教學框架:公司認領 + 派工

每位學生 / 小組**認領園區裡一間公司**,對其生產數據負責。不同公司是不同產業原型
(CNC 廠、半導體廠、AGV 倉、機械手臂線、廠務動力),數據特性各異,學生看見廣度。

---

## 階段一:連線 → 監控 → 處置

1. **連線**:從設備目錄查到該連什麼,**自己寫客戶端**(Modbus / OPC-UA / MQTT 擇一或全做)。
   `student_kit/` 提供最小連線骨架與目錄查詢範例當鷹架,但抓取邏輯學生自負。
2. **監控**:自建儀表板,持續輪詢 / 訂閱 tag,呈現該公司設備狀態。
3. **偵測**:故障由老師注入或退化越界自動觸發 → 學生要**偵測到**異常。
4. **處置**:
   - 確認告警、開 / 結工單(`/api/tickets`)。
   - 進階:**寫入暫存器 / 呼叫 OPC-UA method 下達 reset / 維修命令**,把設備從 fault 拉回。
     這把「只讀監控」升級成「讀寫控制」,更像真實 SCADA,是漂亮的難度階梯。

### 工單 / 票務

故障自動生成工單(device、type、onset)。學生 ack → 處置 → resolve。
系統量 **MTTR**(平均修復時間)、處置動作是否正確 —— 天然的評分指標。

---

## 階段二:分析 → 訓練 → 閉環即時推論

學生把歷史數據(Historian)帶回去訓練故障診斷 / RUL 模型,再把模型**接回園區做即時推論**。

### 閉環架構(已選此方案,非交檔評分)

```
學生模型(在自己機器或 5090 上)
  ├─ 訂閱 ──> MQTT / WebSocket 該設備即時遙測
  ├─ 推論 ──> 預測「該設備將於 eta 後發生 fault_type」
  └─ 回報 ──> POST /api/predictions {device, predicted_fault, eta, confidence}
                    │
                    ▼
         scoring 用 ground-truth 的 fault_onset_time 比對
              → lead_time = fault_onset − prediction_time
              → 設備在 2D 世界翻「預測故障(橘)」狀態
              → 命中事件廣播到 /ws/events
```

**計分預警提前量(lead time)**:在實際故障注入**之前**就預測到 → 得分,提前越久分越高;
誤報(該設備其實沒壞)扣分。這就是預測性維護的完整迴路,既真實又有畫面。

> 「把模型匯入設備做預測」=「學生模型當成一個訂閱遙測、推回預測的服務」。
> 不需要真的把權重塞進設備韌體,閉環效果與教學意義一致。

## 自動評分(ground-truth 在手)

因為有 ground-truth,下列可自動算,對人數多的課是減負神器,也讓排名 / OEE 競賽成立:

| 階段 | 指標 |
|------|------|
| 階段一 | 故障後幾秒內偵測到、處置動作正確性、MTTR、漏報 / 誤報 |
| 階段二 | 預測 lead time、F1、RUL 誤差(RMSE)、誤報率 |
| 綜合 | 該公司 OEE、設備可用率排名 |

## 進階教學模式

- **情境腳本 / 災難日**:預寫連鎖故障腳本(`/api/scenarios/{name}/run`),當期末實作測驗,
  全班同條件。
- **OEE / 排名榜**:公開計分榜,把監控品質、處置效率、預測準確度做成競賽。
- **感測器 vs 設備故障題**:同時注入兩類,考學生根因判斷。
