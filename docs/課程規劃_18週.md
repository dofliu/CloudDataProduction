# 雲端生產數據導論 — 18 週課程規劃

> 本檔是課程與平台的對照藍圖。課程主軸是**資料的一生**,平台(本 repo)是教具兼自動出題/批改系統。
> 隨學期滾動修訂;平台待開發項見文末「平台配套」。

## 課程定位

**這是「雲端生產數據導論」,不是故障診斷分析課。** 主軸是資料生命週期:

```
產生 → 用工業協定接取 → 傳輸/串流 → 儲存 → 分析/統計 → 視覺化 → KPI/決策 → 輕量預警 → 報告
```

故障 / 異常只是「拿來分析的資料素材」,用來練資料處理與判讀,不是教維修診斷。

> **課名視角補充**:「雲端生產數據」可斷成「雲端｜生產數據」(數據上雲,本課主線)與「雲端生產｜數據」
> (雲製造這種生產模式的資料)。後者的概念、與本平台的邊界、以及可討論 / 可動手的延伸議題(含 MES /
> 雲端生產管理),見 [雲端生產_概念與議題.md](雲端生產_概念與議題.md)。建議在 W1 開場與 W11(KPI/MES)帶入。

## 教學模式與節奏

- **一般教室(非電腦教室)**:課堂 = 老師講解每週進度 + 動態展示(投影)。學生課堂用**手機觀察**,不做大量操作。
- **真正動手在回家作業**:學生自己寫 client、自建介面、自己分析。因此作業需可**遠端存取**平台(Cloudflare Tunnel / Tailscale)。
- **每週釋出制**:老師每(數)週在平台設定一個「資料情境」(訂單量、開/關某異常、圈定資料窗),學生把該情境的資料帶回家分析。
- **60 人、無助教 → 批改以自動為主**:能對 ground-truth 自動比對的作業優先;人工評分集中在期中報告與期末專題。

## 修課對象與難度

- **對象**:大三 / 大四選修,具 Python 基礎。連線 / 除錯鼓勵善用 AI 協助(不另附 notebook)。
- **分組**:**期中前個人**、**期末兩人一組**。
- **班級場景**:個人期需人人有一間公司 → 用 `scenarios/class_park.yaml`(**64 間公司**,每間至少 1 台會生產/退化的設備,一人一廠夠用;`SCENARIO_FILE` 切換)。`default_park`(23 廠)留作展示 / 小班。
- **難度分軌**:每份作業含**基礎題**(連線 / 敘述統計 / 明顯異常)+**進階題**(訊號相關、根因判斷、時序聚合、RUL 估計),同時服務跟得上與想挑戰的學生。進階題見「平台配套(2)」。
- **廣度控管**:核心線為「接取 → 分析 → 視覺化 → KPI」。**W12 資料管線、W13 的趨勢/預警(ML)設為選作/加分**,避免導論課廣度過載;想深入的學生可加分,其餘專注核心。

## 學期行事曆(含假期)

- **停課週(國定假日 / 研討會)**:W3、W5、W14、W16、W17。這些週不上課。
- **放假前一週給「整合作業」**,涵蓋停課週:W2(接 W3)、W4(接 W5)、W13(接 W14)。
- **W15 上完直接進期末**:W15 為最後一次授課,同時發布期末專題大目標;W16–W17(停課)為專題自主進行週;W18 期末發表。
- **不是每週都有作業**:作業集中在整合週與關鍵週,其餘為課堂觀察 / 練習。
- **課堂即時練習**:非作業週的「課堂觀察 / 練習」可用內建的即時練習——老師一鍵佈題、學生手機作答、
  即時批改、計入平時成績(比回家作業輕)。見 [課堂即時練習.md](課堂即時練習.md)。

上課週:1、2、4、6、7、8、9、10、11、12、13、15、18(共 13 次;其中 W9 期中、W18 期末)。

## 作業與批改策略(60 人、無助教)

原則:**能自動就自動**。平台握有 ground-truth,學生把算出的數字 POST 上來,系統在容差內自動計分。

- **自動批改**(對 ground-truth 比對):W2 連線成功、W4 週統計吻合、W6 異常設備/時段命中、W11 OEE/產出吻合、W13 預警 lead time、期中的接取與統計項。
- **人工批改**(rubric,量少而精):W8 資料品質短報告、W12 管線檢視、W13 比較報告、W15 提案、期中報告、期末專題。
- 全學期正式繳交約 **6–7 次**(整合①②③、期中、期末提案、期末專題),其餘為課堂練習,控制在單人可負荷。

## 期中評量(W9,個人)

主題:**一台設備的資料分析**。

1. 用**任一協定**成功接取自己設備(自動:平台看得到連線 / 交的即時值對得上)。
2. 一週資料的**敘述統計**,與 ground-truth 比對(自動:落在容差內給分)。
3. **自建儀表板**呈現(人工 rubric)。
4. 比較「正常週 vs 明顯異常週」的資料特徵並書面說明(人工 rubric)。

配分建議:自動 40% + 儀表板 30% + 報告/洞察 30%。

## 期末專題(W15 啟動 → W18 發表,兩人一組)

主題:**端到端雲端生產數據應用**(針對自己公司)。

- 多設備接取 → 自有儲存 → 分析/統計 → KPI/OEE 與預警儀表板 → 書面報告 + 口頭發表。
- 評分:自動(資料正確性、系統可動)+ rubric(完整度、分析深度、視覺化、報告與台風)。
- 兩人分工需在報告載明。

## 平台配套(需開發,支撐上述課程)

1. **每週情境釋出(教師面)**:一鍵設定當週訂單量、開/關指定異常、圈定資料窗;可重現(W5 全正常、W6 明顯異常、W8 感測器故障…)。**最高優先**。
2. **作業自動比對(繳交端點)**:學生 POST 統計/KPI/預警,平台以 ground-truth 容差自動評分(W4/W6/W11/W13/期中直接受惠)。
3. **資料窗 / 週次標記**:讓「這週的資料」有明確 sim_t 起訖,作業可界定、可重現。
4. **遠端存取**:Cloudflare Tunnel(HTTP)+ Tailscale(原生協定),作業在家做的前提。

> 建議先做 (1)(2):平台即從 sandbox 變成「每週自動出題 + 自動批改」的教具,正中「一般教室 + 手機觀察 + 回家作業」的模式。

### 已實作(1)(2)(3):使用方式

**(1) 每週情境釋出** — 定義於 `scenarios/course_weeks.yaml`,教師手動套用:

| 端點 | 權限 | 說明 |
|---|---|---|
| `GET /api/course/weeks` | 公開 | 列出各週情境(異常/密度) |
| `GET /api/course/status` | 公開 | 目前第幾週、資料窗起點、稼動率 |
| `POST /api/course/weeks/{n}/apply` | 教師 | 套用第 N 週(設異常、稼動率、記資料窗起點) |

- `faults`: `clear`(全部修復正常)/ `keep`(沿用)/ dict(每間公司挑 count 台注入設備或感測器異常)。
- `order_density`: `light/normal/high` → MES 稼動率 0.55/0.80/0.98(改變待機比例、退化、OEE)。

**(2) 作業自動比對** — 學生 `POST /api/submissions`,平台對 ground-truth 容差計分。**每份作業含基礎題 + 進階題**,一次照顧全班分佈:

| type | 層級 | 對應週 | payload | 比對對象 |
|---|---|---|---|---|
| `connect` | 基礎 | W2 | `{student, device, tag, value}` | 平台當下實值 |
| `stats` | 基礎 | W4/期中 | `{student, device, tag, metric:mean\|std, value}` | Historian 當週資料窗真值 |
| `anomaly` | 基礎 | W6 | `{student, devices:[...]}` | 本週實際異常設備(F1) |
| `events` | 基礎 | W10 | `{student, device, value}` | MES 完工工單數 |
| `aggregate` | **進階** | W7 | `{student, device, tag, hour, value}` | 依 hour-of-day 重取樣的平均 |
| `correlation` | **進階** | W8 | `{student, device, tag_a, tag_b, value}` | 兩訊號皮爾森相關 r(自撈兩序列、對齊、算相關) |
| `root_cause` | **進階** | W8 | `{student, device, cause:sensor\|equipment}` | 感測器故障 vs 設備故障(對錯,不透露細節) |
| `oee` | **進階** | W11 | `{student, device, metric, value}` | 平台 OEE 累積器 |
| `rul` | **進階** | W14 | `{student, device, value(小時)}` | 隱藏 RUL ground-truth;**正解不公開**,只回饋在容差內與方向 |

進階題刻意有難度:`correlation` 要自己撈兩條序列對齊、`root_cause` 要分辨資料品質 vs 真退化、`rul` 得從趨勢外推且看不到正解。
`GET /api/submissions?student=&week=&type=`、`/leaderboard`、`/gradebook` 供查詢 / 排名 / 成績冊。

**(3) 資料窗**:套用週情境時自動記錄 `window_start_sim_t/wall`;`stats` 作業預設以「當週資料窗起點→現在」為範圍(亦可在 payload 帶 `from`/`to` 覆寫)。

**(4) 遠端存取** 仍待部署(Cloudflare Tunnel / Tailscale)。
